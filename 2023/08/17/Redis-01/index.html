Reids实现的锁

在设计Redis分布式锁时，根据锁的应用范围，设计锁在Redis实现中的key值，即理清要求串行执行是哪类线程。假如在限制「每人一单」的场景，key应该随用户id而变。

在极端情况下，Redis分布式锁会发生误删除锁的情况：线程1占有锁(Lock:Thread1)，在完成任务时阻塞，锁过期被释放，这时线程2占有锁（Lock:Thread2），随之线程1完成任务释放了线程2占有的锁(Lock:Thread2)。
解决：

  1. 锁的值存储线程标识。这个标识不能单纯是线程id，因为这样仍然保证不了唯一性——不同JVM进程下，各自线程id的分配是私有的。集群下会出现不同JVM的线程之间id相等的情况。因此使用UUID作为前缀，创建UUID变量作为分布式锁这个类的类属性。这样，不同JVM有着不同的UUID，同一JVM可以靠线程id进行锁的区分。
  2. 在释放锁时，判断是否是当前线程所占有的锁，结果一致则释放，反之不释放。判断与释放是先后的两个动作，这之间如果线程阻塞，锁过期并有其他JVM下的线程获取锁，那么在前线程结束阻塞执行释放时，仍然错误地释放了其他线程的锁。因此，需要确保“判断-释放”这个过程的原子性——使用lua脚本解决。lua脚本可以实现将代码逻辑与redis命令调用写在一起，实现原子性。

Redisson锁的看门狗机制：一个线程获取锁，开始执行业务代码，只要线程还在执行业务代码，那么锁的过期时间每过10s就重置为30s（这是一个可以递归调用的定时任务）。只要业务代码执行尚未执行完，锁就不会过期。解决了业务代码阻塞，锁自行过期，导致其他线程也进入同一业务代码的问题。在主动释放锁时，会取消这个定时更新的任务。

Redisson分布式锁的特点：

  • 可重入：利用hash结构记录线程id和重入次数
  • 可重试：利用信号量和Redis发布订阅功能实现等待、环形，获取锁失败的重试机制，避免了直接忙等待。
  • 超时续约：利用watchdog,每隔一段时间（releaseTime / 3），重置超时时间。
  • 避免锁的主从一致性问题：multiLock。为避免锁的主从一致性问题，在获取锁时，Redisson将锁信息保存在所有Redis节点中。在所有节点都获取到锁，tryLock()才算成功。（此前一般是只在主节点进行写操作的）。这点感觉不算是很好的解决方案。 但Redisson源码写得好严谨！！

Redis实现的消息队列

使用jdk的阻塞队列存储待持久化的订单信息，有如下缺点：

  1. JVM有着内存上限的限制
  2. JVM的内存没有持久化机制，服务重启或宕机会使JVM内存中的订单信息丢失，无法找回。以及队首被取出后处理的过程中出现异常，导致该队首信息丢失。

Redis消息队列：
Redis Stream(XGROUP)消息队列是一种比较完善的消息队列。生产者使用XADD命令向指定stream中添加消息；消费方创建消费者组XGROUP CREATE，使用消费者组读取消息XREADGROUP，确认消息XACK。对于每个消费者组，服务维护一个pending list，存放该消费者组读取但尚未确认的消息。因此使用XREADGROUP命令时，可以指定消息的读取来源是stream (>)，还是pending list (0)。
命令比较多，参数繁杂，需要善用redis help命令和redis官方文档。
综上，这种消息队列实现方案的优势有：

  • 消息可回溯。同一条消息不能被同组的消费者都读取，同组的消费者属于竞争关系；但可以被不同组再次读取。
  • 可以多消费者竞争消息，加快消费速度。
  • 可以阻塞读取。在命令中设置BLOCK参数。
  • 没有消息漏读的风险。每次消息读取可以指定读stream中首个待读取的消息。
  • 有消息确认机制，保证消息至少被消费一次。因为每个消费者组对应一个pending list存放该组读取过的待确认消息。

Redis实现的一些业务

关于「点赞与取消点赞」业务：（假设这里点赞的内容是某篇笔记）
点击点赞按钮时，前端可以根据按钮的亮暗状态（亮为已赞，暗为尚未点赞）切换状态即可。但只是这样做并不够，因为在显示笔记时就需要正确显示「是否已点赞」，尽管用户没有主动操作按钮。因此但涉及笔记查询业务的地方，都需要前端正确判断该篇笔记「是否已点赞」，这就必须有后端提供的信息，根据信息判断。一种做法是为笔记添加Boolean isLiked字段，表示当前登陆的用户是否赞过这篇笔记。在需要显示笔记的情形下，后端查询笔记实体，包装好必要的信息返回给前端，包括了笔记的isLiked字段，前端根据该字段显示已点赞情况。

feed流中的推拉结合：
“拉模式”存在可能较大的读取延迟，“推模式”存在可能较大的内存占用。
普通用户粉丝少，使用「推模式」，分布动态时将动态推送到每个粉丝的收件箱，这样避免高读取延迟并使用了较少内存。
大V粉丝多，但包括活跃粉和普通粉：活跃粉占比少，使用「推模式」推送消息；普通粉僵尸粉占大多数，使用「拉模式」，在粉丝用户读取收件箱时从关注的大V的发件箱中拉取动态。

滚动分页查询使用redis的zset实现而不是list,根据score值来分页查，避免使用list造成的新添加数据时索引变化引起的某条数据重复查询。
滚动分页查询参数：

max：第一次查询：当前时间戳；后续查询：上一次查询结果中的最小时间戳。
min：固定为0。
offset：第一次查询：0；后续查询：已查询到的所有结果中score值为最小时间戳的元素个数。
count：固定值，为每页数据条数。

Redis使用注意事项 - 内存方面

redis key 限制在44字节大小，因为key是string类型，redis底层编码包含int, embstr, raw三种，在44字节内使用embstr，采用连续内存空间；超过44后使用raw编码方式，具有内存消耗较大和访问性能下降的弊端。

hash结构的entry数量不要超过has-ziplist-max-entries，这个值可以通过config get hash-max-ziplist-entries命令查看，也可以通过config set设置。超过这个值，底层存储方式将不再使用更节省内存的ziplist。
