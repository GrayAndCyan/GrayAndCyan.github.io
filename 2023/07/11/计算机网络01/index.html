分层的体系架构

什么是网络：节点与边构成大小形状无关的拓扑
什么是计算机网络：联网的计算机所构成的系统。节点有主机节点（方），数据交换节点（圆）。主机节点是数据的源/目标。连接这些节点的是链路。边分两种：与方的有关——接入链路；连接两个圆的，骨干链路。
互联网Internet：由一簇协议支撑的规模最大的网络

网络交换设备：是数据的中转节点，数据交换节点。路由器工作在网络层，交换机工作在链路层

端系统 end system / host

协议：对等层的实体（peer entity）在通信的过程中应当遵守的规则的集合。包括内容为格式、次序和动作。

格式：PDU（协议数据单元）

次序：通讯需要在某个动作之后发送，例如web服务器在收到请求后发送http报文。

互联网构建方式：群策群力——向IETF提交RFC文档并公开评头论足

从服务角度，互联网是分布式的应用进程以及为分布式应用进程提供通信服务的基础设施。分布式的应用是网络存在的理由。基础设施向应用进程提供服务的方式是API的方式。提供的服务有面向连接的TCP和无连接的UDP。

网络边缘

互联网包括三个子系统：

  1. 跑了网络进程的主机组成边缘系统
  2. 网络交换节点与链路为网络的核心。作用是数据交换
  3. access——边缘接入核心，「接入网」，「接入系统」

应用进程通信的两种模式：

  1. CS模式

    存在可扩展性差，请求数量增加到某个阈值时，服务器的能力断崖式下降；以及存在可靠性问题。

  2. P2P模式

    每个节点即使客户端，又是服务器。

基础设施向网络应用提供服务的两种方式：

  1. 面向连接的通讯方式

    TCP
    可靠，不冲突，不失序，不出错
    能自动根据协议协调发送的速度和接收的速度，以至于避免发送方发送速度过快远超于接收方处理能力。
    拥塞控制特性：感知路径中通断/通畅的情况
    花费很多时间的代价，不适合实时多媒体应用

  2. 无连接的通讯方式

    UDP
    不可靠
    没有流量控制、拥塞控制
    发送速度与产生速度一致，不加控制，速度快，适合网络实时多媒体应用

网络核心

  • 电路交换/线路交换：如电话网。连接线路不共享，作为两通信主机间专有的资源，保证性能，同时存在资源浪费的问题。要求建立呼叫连接，需要时间成本。由于上述资源浪费和时间成本，线路交换不适合计算机网络通信
    将带宽分成片：频分（FDM），时分（TDM），波分（WDM）
    核心交换机一旦宕机，影响范围十分广

  • 分组交换：存储-转发。可变长排队延迟，可能出现丢失。按需使用，资源共享。

有连接的虚电路网络，虚电路表
无连接的数据报网络，路由器

Internet结构和ISP

全连接：不可扩展——n^2,代价过大

ISP：Internet Service Provider 互联网服务提供商 西电 、陕西移动、陕西联通 提供接入，提供网

丢弃怎么办？如果数据链路层可靠（无线链路，wifi），由上一个节点重传，如果不可靠（以太网），由源端主机重传（TCP），如果应用进程通过UDP放出IP分组，不重传。

协议层次和服务模型

一个形象的例子表明计算机网络的模块分层结构：异地不同语言的哲学家进行实时思想沟通：哲学家层-翻译层-秘书层，秘书层通过下层提供的服务，如电信公司提供的服务、互联网提供的服务、马车…将信件交给对方的秘书层。秘书层解决异步通信的问题，翻译层解决表示转换的问题哲学家层相当于应用层，借助下层服务完成哲学思想的交流。两个翻译需商量好用一个共用语言，这就是协议规范的内容；秘书采用的通信方式也是协议的内容。

如此分层的好处：

  1. 把复杂问题转换成几个子问题，子问题的解决相对独立和单一；
  2. 下层实现对上层透明，便于采用新技术，平滑升级。可以从协议的进化上体现。
  3. 便于交流与讨论。

问题：效率较低

🔺更正式的描述，关于计算机网络分层：把网络非常复杂的功能分解为若干个功能明确的层次，每一层实现一个或一组功能，上层的模块通过层间接口使用它紧接的下层的服务。服务是功能的一部分，是外部能通过接口使用的功能的一个子集。对等层实体的模块通过协议交换控制信息。协议是对等层的实体在通信的过程中应当遵守的规则的集合，协议的目的是向上层提供更好的服务，协议通过层间的接口访问下层所提供的服务才能实现，这是协议与服务的关系。最底层的物理层不借助任何一层的服务，直接向对方发送每个bit的电磁波信号或光信号。

服务访问点（Services Access Point）：层间界面上用于下层的服务提供者区分不同的上层用户的一些信息，例子：邮箱，传输层的SAP是端口Port

原语（primitive）：服务提供与使用的形式，如Socket API

面向连接的服务：两个用户在使用下层所提供的服务之前，有“握手“的关系，为后面通信做准备。如TCP

无连接的服务：不握手。如UDP

服务数据单元SDU：上层要求下层所传输的数据。

SDU + ICI（接口控制信息）形成本层的PDU（协议数据单元），本层的PDU又作为下层的SDU。


类比卡车头与货物

每一层的PDU称呼不同，例如应用层-应用报文message，传输层-报文段segment…

第二章 应用层

应用层原理

按照多个进程之间的交互方式分为以下架构：

  1. C/S模式
  2. P2P模式
  3. 混合模式

网络层提供主机的标识和寻址的方式是IP地址
但一台主机上运行了多个应用进程，于是…
传输层使用端口号Port用以区分不同的应用进程，TCP和UDP的端口号都是16bit，但TCP和UDP使用端口号的方式不同

数据传输需要三方面信息：谁发 谁收 数据本身
谁发和谁收都要主机IP与端口号两个信息标识
两端需要连续通信较多的数据，每次发送接收都要传输以上三点信息，显然“谁发”“谁收”在两端的后续传输中是冗余的，如何减少层间传输的信息量？采用Socket是解决方案。
Socket实际是一个整数，是一个代表会话关系（TCP Socket）的本地标识。当与对方建立起一个连接，建立起通信关系，Socket会返回一个标识这个连接的整数。这个整数标识了“谁发”“谁收”信息，即本进程的ip+端口号，以及对方进程的ip+端口号，这个整数代表了这样的一个四元组，这是TCP的情形；UDP情形下该整数则是代表个二元组，只包含自己的ip+端口号。
之所以说是本地标识，不是全局标识，是因为Socket整数值及其所映射的内容这一映射关系，只有本机操作系统知道，通信的对方并不知道，对方有对方的Socket。
采用Socket不仅减少了每个会话中层间传输的信息，而且简单，便于操作系统管理。

UDP的Socket是一个本地意义的标识，标识本机ip和进程端口号信息。穿过层间接口的信息大小最小。但在传输报文时，必须指明对方IP、port；接收报文时，传输层需要上传对方的IP、port。

只从应用层视图上看，Socket就像两通信进程间的传送门，多啦a梦的任意门。

SSL：安全TCL

TCP和UDP都没有对数据的加密，以明文形式通过互联网传输。
SSL：运行在TCP之上的安全套接字层，提供加密的TCP连接。通常采用库的形式与应用进程一起运行。提供SSL级别的Socket API，应用通过API将明文交给SSL Socket，SSL将其加密在互联网上传输。
https——web应用不是直接运行在TCP之上，而是运行在SSL之上，SSL over TCP。
SSL提供私密性，完整性，鉴别或可认证性

Web和HTTP

TCP向上层提供的服务不提供报文间边界，应用进程需要区分报文间边界。

http是无状态的协议，即服务器不维护客户端的状态。

在需要维护客户端状态的情景下http协议变得不实用，因此引入Cookie。

web浏览器在第一次向某服务器发送请求时，通常不带Cookie。服务器为其创建Cookie并保留到数据库，在respond中返回给客户端。客户端收到cookie后也保存在文件系统。下一次访问，在请求报文中包含cookie，服务器得以辨识，得到客户端状态。

cookies的应用：

  • 用户验证
  • 购物车
  • 推荐
  • 用户状态（Web e-mail）

cookie带来隐私性问题，它会暴露网站访问行为与信息，将其关联可以挖掘到用户更多的信息。

Web缓存
不访问原服务器，而是访问代理服务器（proxy serve），代理服务器作为缓存。当请求的对象不在代理服务器时，即缓存未命中，缓存请求原服务器该对象，并返回该对象给客户端。

好处：对客户端，减少响应时间；对原服务器，减少请求数，降低荷载。

如果缓存（代理服务器）加在局域网，缓存命中，即实现了对象的本地访问；否则需访问外网，为远程访问。

条件式获取（conditional get），解决缓存对象与原服务器一致性问题：在缓存向服务器发出的请求的头部行中添加if-modified-since字段，服务器根据修改与否的判断结果，返回未修改通知（304 Not Modified），或改动后的新对象。

e-mail

3个主要组成：

  • 用户代理
    又名邮件阅读器，是撰写/编辑与阅读邮件的客户端软件，其输出和输入的邮件保存在服务器上。通过用户代理软件来访问电子邮件这个应用，可以说这个软件就是电子邮件应用的代理。比如，浏览器是web应用的用户代理。
  • 邮件服务器
    报文队列持有待发送邮件的报文
    邮箱持有接收方的邮件
  • SMTP（简单邮件传输协议）
    用于邮件服务器之间的邮件传输，服务器守护在25号端口。
    传输分为三个阶段：握手，传输报文，报文。
    命令/响应报文全部在ASCII码表示范围内。ASCII码用7位表示128个字符和控制符，用1B表示一个ASCII码时最高未通常固定为0。

发送邮件：用户代理发给邮件服务器，排在报文队列中，邮件服务器把队列中的邮件发给目标邮件服务器，目标服务器将邮件放到相应的邮箱中，前两跳传输协议是SMTP。接收方用户代理通过POP3、IMAP、HTTP等存取协议从位于相应邮件服务器的邮箱中，把邮件拉取（下载）到用户代理软件，供接收用户阅读。

MIME（多媒体邮件扩展），规定格式，实现通过SMTP协议传输多个对象，内容使用Base64编码，因此对象的每个字节都可以不再ASCII码描述范围内。Base64可以将若干个不在ASCII码表示范围的字节映射为更长的但每字节都在ASCII码表示范围的结果。
如：
转换前：10101101,10111010,01110110
转换后：00101011,00011011,00101001,00110110

提供域名到IP地址的转换服务。

主要思路：

  • 分层的域名
  • 分布式的解析和维护

一台设备上网必须要具备的四个信息：

  1. 主机的IP地址
  2. 所在子网的子网掩码

  • 目标名字在Local Name Server：查询的名字在该区域内部，或是缓存。

CDN

####多媒体流化服务：DASH

DASH: Dynamic, Adaptive Streaming over HTTP 动态自适应HTTP流化

简单场景：

  • 服务器：
    • 将视频文件分割成多个块
    • 每个块独立存储，编码于不同码率（8-10种）
    • 告示文件（manifest file）: 提供不同块的URL
  • 客户端：“智能”客户端，客户端自适应决定何时去哪请求什么编码速率的块
    • 先获取告示文件
    • 周期性地测量服务器到客户端的带宽
    • 查询告示文件,在一个时刻请求一个块，HTTP头部指定字节范围
    • 如果带宽足够，选择最大码率的视频块
    • 会话中的不同时刻，可以切换请求不同的编码块 (取决于当时的可用带宽)

CDN（Content Distribution Networks） 内容分发网络

“让内容靠近用户～”

少量服务器面对高并发情形存在的问题：

  • 跳数多，瓶颈链路制约吞吐
  • 网络上重复流量多，资源浪费严重
  • 单点故障点，可靠性差
  • 服务器群周边网络拥塞

CDN解决高并发：
ICP购买CND服务。CDN运营商在全球范围部署网络缓存节点，将内容预先部署在缓存节点上，用户访问时通过域名解析的重定向，定向到对其而言服务质量最好的缓存节点，这些节点提供相应的内容服务，即内容加速服务。

CDN运营商部署缓存服务器的策略：

  • enter deep：部署在Local ISP附近，质量高，但数量高（成本up）
  • bring home：部署在少数关键位置，如同时距离多个上层ISP较近的位置。
    深入群众与关键少数～

CDN实现在应用层、网络的边缘，实现加速访问——“Over The Top”

例子：netflix

TCP Socket编程

TCP：可靠的、字节流的服务，不保证报文的界限，需要应用维护报文间界限

UDP：不可靠的、无连接的（不握手）的服务

Server：

  • 服务器先创建一个WelcomeSocket

  • bind，绑定本地的一个端口号到WelcomeSocketId，使WelcomeSocket守护本地该端口

  • accept()方法，阻塞式等待来自客户端的连接，接收到来自客户端连接后解除阻塞。由WelcomeSocket调用，解除阻塞后返回新的Socket，（connectionSocket），作为所成功建立的连接的会话标识。即connectionSocket同时表示了本地信息与对方信息，代表有效的连接。

  • read request from connectionSocket

  • write reply to connectionSocket

Client：

  • 创建Socket，隐式地捆绑SocketId与本地IP+端口
  • connect()方法，需要指定服务端主机IP与端口号，连接到这个服务。（显式地捆绑对方的ip+port）。与对方建立连接，对方给出应答，此方法方可解除阻塞。
  • send request using clentSocket
  • read reply from clientSocket

UDP Socket编程

每个Socket只标识本地信息，因此在发送消息时不仅要包含Socket与消息内容，还要加上对方ip+port （发给谁）。收报文时，需要收到消息内容与对方端节点两方面信息。

传输层

概述和传输层的服务

传输层的服务：进程和进程之间的、以message（报文）为单位的、逻辑的通信。

网络层IP提供的服务是主机到主机。

加强服务品质，把网络层的不可靠性加以弥补。但有些服务品质是加强不了的，例如延迟和带宽。

实现在源端复用，目标端解复用。（聚合与拆分）

流量控制，拥塞控制，连接管理

TCP提供可靠的、字节流的服务，UDP提供不可靠的、数据报datagram的服务。

RDT 可靠数据传输

采用渐增的方法发展RDT协议

RDT1.0：封装与解封装

RDT2.1：发送方发送Packet，接收方收到Packet后检错，根据检错结果响应ACK（没问题）或NAK（出错了）给发送方。如果收到ACK，发送方发送下一个Packet，反之重复发送旧的Packet。因此，如果ACK/NAK本身出错，发送方仍发送旧的Packet；接收方重复收到该Packet，表示上一次受到的Packet是正确的但ACK传输出错，为获取新Packet，再次响应ACK。同样地，「重复发送」这个机制也完美解决了NCK传输出错的情形。
“停止等待协议”
RDT2.0是除去重传方案的RDT2.1

RDT2.2：无NAK的协议
功能同rdt2.1，但只使用带编号ACK，编号是最后正确接收的分组的序号。
当收到重复的ACK（如：再次收到ack0）时，发送方与收到NAK采取相同的动作：重传当前分组。
为后面的一次发送多个数据单位做一个准备。

RDT3.0：具有比特差错和分组丢失的信道。

流水线协议
流水线：允许发送方在未得到对方确认的情况下一次发送多个分组，提高链路利用率。
两种通用的流水线协议：回退N步(GBN)和选择重传(SR)
对比：

GBN SR
优点 简单，所需资源少（接收方一个缓存单元） 出错时，重传一个代价小
缺点 一旦出错，回退N步代价大 复杂，所需要资源多（接收方多个缓存单元）
适用范围 出错率低：比较适合GBN，出错非常罕见，没有必要用复杂的SR，为罕见的事件做日常的准备和复杂处理 链路容量大（延迟大、带宽大）：比较适合SR而不是GBN，一点出错代价太大

连接管理

连接是谁的连接？
连接是两个应用进程建立起TCP的连接

连接建立的本质：

  • 双方知道要和对方通信
  • 为通信准备相应资源。如发送缓冲区，接收缓冲区。
  • 控制变量置位，一些控制变量需发送给对方并得到确认

3次握手

第三次握手往往与数据传输在一起。

只采用两次握手带来的问题：

  1. 服务器维护的虚假的半连接，资源浪费；
  2. 在半连接的情形下，服务器错把重发的旧数据当作新数据。

三次握手解决上述问题：

  1. 由于存在第三次握手确认，客户端可以辨识出虚假半连接并拒接。解决问题1。
  2. 不存在半连接，并且有着初始序号的限制作用，如果在原连接结束后，或新的连接建立，服务器不接收重发的旧数据。解决问体2。这也解释了「初始序号不能固定为常数，而要随机」的原因。

两个应用进程之间的TCP连接分成两个方向，每个方向单独拆除。每次拆除需要一方发送拆除请求，另一方确认。经过两次交互，并不完美，存在两军问题，导致一方维持着连接 另一方已把连接拆除的情况。

拥塞控制原理

拥塞：太多的数据需要网络传输，超过了网络的处理能力。拥塞时表现为网络延迟高、丢包率高。

路由： 规划源主机到目标主机的详细路径。 控制平面功能 全局的功能 计算路由表
