  1. 如何理解IoC
    首先，IoC本质是一个轻量级容器，使用ConcurrentHashMap存放各业务对象，这些对象我们称之为Bean。IoC 管理这些bean的生命周期，比如读取bean定义并注册，进行实例化以及完成依赖注入。
    《Spring揭秘》所说，***IoC是一种解耦各业务对象之间依赖关系的对象绑定方式。***
    关于解耦依赖关系：试想没有IoC的场景，对象A使用new的方式直接依赖对象B，是紧耦合。IoC的解耦作用体现在有了IoC后，对象A只需要去IoC容器中拿到B对象，就可以去使用它，也就是IoC帮我们完成的依赖注入，在需要更改A依赖的对象时，不需要改动A类的代码，免去重新编译A类源码，只需要更改配置文件或配置类即可，达到解耦效果。
    而对象绑定，正是在前面所说的，IoC帮我们完成依赖注入。根据依赖属性的类型分两种情形：对于依赖属性是普通对象，ioc使用反射技术，在对象实例化后注入属性；对于依赖同样是bean的情况，spring在读取bean定义时，使用BeanReference类型进行包装、标识，ioc根据BeanReference拿到依赖的beanName，去在容器中获取这个bean对象实例，给当前bean装配上；如果容器中没有我们需要的这个bean的实例，会先去完成这个新bean的实例化与初始化。

JavaSE

  1. 为什么String类要做成final的？

    安全性：

    final 使 String 类不可被继承，从而避免了子类对String类「不可变」这一特性的修改，避免了因此导致的线程安全问题。
    String在Java中发挥举足轻重的作用，不仅经常作为数据的键值，还广泛用于表示一些敏感信息，如密码、密钥，所以保证这个类的安全性这点毋庸置疑，尽管为此牺牲了可继承的这种灵活性。
    那么具体地，为什么加final可以保证安全性？String类是不可变的，尽管一些方法看似改变了字符串，但实际是拿到了新创建的字符串，原字符串对象没有改变。当多个线程对共享资源读写操作时，由于读写操作非原子性以及CPU不断进行着线程切换，容易造成线程交错执行导致线程安全问题，刚才提到的字符串不可变特性正避免了这个情况的发生，由于不可变，所以线程可以读它，但有不了写操作，因此String是线程安全的。了吗？不完全，如果它可以被继承，那么子类就可以修改它的方法，破坏原本特性，最直接的做法，比如子类方法中创建新线程去操作了原有资源包括局部变量和成员变量。添加final修饰符杜绝了这一可能性。对修改关闭，这也符合开闭原则。

    总之String需要保证线程安全，它本身的不可变性为之增添了一层保障，而final是 维护其不可变性，巩固线程安全性 的又一层保障。尽管它让开发者无法继承String类来编写自己的字符串类，但这一取舍是值得的。我是这样理解Java设计者的这种设计的。

    （回答思路：线程安全的必要性 -> 自身不可变性使线程安全 -> 加final不可被继承使本身特性不被破坏）

    性能方面：

    字符串的不变性既避免了读写冲突，又能在性能方面有所贡献。每个String对象可以缓存根据字符串内容计算得到的hash值。这样每个字符串对象的hash值只被计算一次而无需重复计算，可以用于快速比较字符串的相等性——先比较hash值，hash不同则可以直接返回false。

  2. 说说 synchronized 锁升级的原理

synchronized锁升级在jdk15前后有一个较大的改变， 我先说下在jdk15之前的做法，主要就是对偏向锁的应用。jdk15之前对象还是默认可偏向的。在执行到synchronized语句时，如果这个对象此前没有被加锁，也就是当前线程获得加锁范围代码的执行权，这时候对象的的markword就会存储一个这个线程地址的信息，表示这个对象现在偏向这个线程。当再有线程想要获取锁时可以快速判断有没有发生竞争。如果再来的线程是新一个线程，根据是否发生竞争选择锁升级为轻量级锁还是重量级锁。

当不发生竞争时选择升级轻量级锁，对象头markword后两位改为00表示加了轻锁。在线程栈中创建一个锁记录lock record，锁记录存储一份加锁对象原本的markword信息和对象引用，然后使用原子操作cas将加锁对象markword剩下的位置存上这个锁记录地址。加锁期间再有线程来向对象获取锁，能根据对象头存的锁记录地址判断是加锁线程锁重入情况，还是来了新线程的竞争情况。当发生锁重入时会在线程栈新创建lock record存加锁对象引用，但不再存储加锁对象原markword信息，而且改为存空指针实际上是零值。

当发生竞争时锁升级为重量级锁。重量级锁的做法是在操作系统层面的monitor对象，通过对象存的lock record地址拿到对象原markword信息存到monitor。monitor有三个主要字段，owner存当前线程地址，表示重锁主人是这个线程；entryList存放阻塞在锁等待争抢锁的线程们，waitSet存调了wait() join()处在等待状态的线程。加锁对象markword后两位改为10表示重锁，前面的位置通过cas改为monitor地址。现在重锁加锁完成，owner线程执行临界区代码，执行结束后在entrylist竞争出新的主人线程，没有新主人的话进行monitor撤销和对象头恢复的工作。

jdk15之后不推荐偏向锁并宣布即将废除它，默认关闭，我在jdk21源码中已经看不到偏向锁代码。那加锁流程和上面所说的相比去除了偏向锁的部分，从轻量锁开始，后面部分一致我就不重复建设了。废除偏向锁的原因有两点，一是对于现在的java程序，它对性能不再有明显提升甚至成为拖累者，二是它对jvm其他组件有代码侵入，代码难以维护。

  1. 线程进程区别
    线程和进程是不同层次的、非并列关系的两个概念，所以回答线程与进程的区别，先要明晰两者的概念。一个进程就是一个正在执行程序的实例（比起程序，它更是一个活动，有执行主体、程序、输入、输出和状态），每个进程有其独立的地址空间和执行程序的实体，这个执行程序的实体也就是线程。进程用来把资源集中到一起，线程供CPU调度来执行程序。但如果进程和线程是一对一的关系，那我认为也就没必要再创造出线程这个概念，所以探讨线程离不开多线程，一个进程下通常会有多个线程，我认为一个进程并行运行多个线程，就是对一台计算机并行运行多个进程的模拟，前者是多线程共享进程的地址空间和其他资源，后者是多进程共享这台计算机的物理资源。
    （插一嘴，多线程的意义是通过拆分简化程序设计和提高性能，其中提高性能在于线程比进程更轻量级，更快创建与摧毁，以及充分利用CPU资源）
    刚提到每个进程有独立的地址空间，因此进程之间保持了较高的独立性，通常互不相干；而进程的不同线程之间独立性不高，它们有完全一样的地址空间，共享着进程的资源比如全局变量，可以合作完成一些任务。当然，每个线程有自己私有的程序计数器、寄存器、堆栈和状态，这些东西帮助线程更好地执行程序和上下文切换。

  3. 死锁是什么？形成的条件？
    用语言表述的话就是，有这样一个进程集合：这个集合中的每个进程都在等待资源的释放，但这些资源又被集合中其他进程占有，这么进程集合就是死锁的。
    死锁形成的四个条件：

    1. 互斥条件：资源只有被占有和可用这两个状态。
    2. 占有和等待条件：已经得到某个资源的进程还可以去请求新的资源。
    3. 不可抢占条件：资源被分配后不能被其他进程抢占，只能由持有进程显式释放。
    4. 环路等待条件：把所有进程和资源都看作一张有向图的节点，进程P需要获取某资源R,那就把P指向R；资源R被进程P占有，那就是R指向p。这张图上形成了环。

代理模式、装饰模式、适配器模式，傻傻分不清？

代理模式与装饰器模式的区别在于：谁持有引用，或者说用户是否持有原对象的引用。
试想大量应用装饰器模式的 Java IO 中的类，被装饰对象由用户持有引用并将其传入装饰器；而在代理模式，代理类目的就是对外屏蔽被代理对象，被代理对象是被代理类持有、使用和增强的。

两者都是对原对象的增强。

适配器模式与上面两个模式的区别在于：

  1. 为了达到适配目的，它一定是改变了被适配对象的接口定义的。在装饰器模式中并不强调接口与原对象的一致性，但在代理模式中，代理类是提供与被代理对象完全一致的接口的，让用户就像直接调用原对象一样地调用了代理对象。
  2. 适配器模式只做适配，不做功能增强或改变。目的不同，适配器的目的是适配，它不干扰被适配对象本该提供的功能。手机接口需要的是 Type-c 插头，我耳机是圆孔插头，那转接头（适配器）要做的只是接受圆孔插头， 对外提供 形为 Type-c 的服务，不干涉我耳机的原本功能。这在 Java 编码中体现为适配器类实现 Type-c 接口（定义了用户端需要的规范），实现 offerTypeC() 方法，供用户端调用。

Spring 的 DisposableBeanAdapter 类是使用了适配器模式的一个例子：

被适配对象是 beanDefinition ，调用方想要只是调用 destroy() 就完成「执行 bean 销毁方法」的工作。beanDefinition 提供的不是destroy() 这个接口，而是 destroyMethodName ，那么由 DisposableBeanAdapter 去做「持有功能提供者，统一对外接口」的工作。

那么这里为什么不在 BeanDefinition 中实现 destroy() ，然后调用方去调它的这个方法？

因为 Spring 提供多种 用户定义并指定bean销毁方法 的渠道，不止 “配置 destroyMethodName ”这一种，还有让 bean 类型实现 DisposableBean 接口的这种方式。这是不能在框架的调用处写死的，那不如在调用处只是调用 DisposableBeanAdapter # destroy(Object bean, BeanDefinition beanDefinitio)，而不必关心更多细节。
